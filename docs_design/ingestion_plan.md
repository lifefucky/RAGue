# План Сбора Данных

## Цель

Логика сбора информации с источника (в MVP - только confluence).

1. Обращаемся в Qdrant и по метаинформации находим максимальную дату актуальности документа.
2. Подключаемся к Confluence по SDK и находим в необходимом домене страницы, которые более свежие, чем максимальная дата актуальности документов в Qdrant.
3. Парсим страницы и записываем в Qdrant.

> **Архитектурное решение бота:** датой актуальности документа считаем `updated_at` страницы в Confluence. Дополнительно храним дату ingestion-а, чтобы отличать изменение документа в источнике от момента, когда система его загрузила в Qdrant.
> **Ответ на рецензию:** `updated_at` страницы в Confluence, но будем и хранить дату ingection

## Metadata Attachments

Attachments страниц наследуют метаданные страницы, на которой выложены.

> **Архитектурное решение бота:** в текущем MVP attachments не записываем в Qdrant. Если при обходе Confluence встречаются вложения, сохраняем локально только по одному примеру каждого расширения для анализа будущих конвертеров. Например, если встретились `doc`, `docx`, `pdf`, `pptx`, локально сохраняем четыре файла - по одному на каждое расширение.

> **Рецензия бота:** стоит добавить к attachment собственные поля: `attachment_id`, `attachment_title` или filename, `attachment_media_type`, `attachment_version`, `attachment_updated_at`, `parent_page_id`. Наследование metadata страницы полезно, но без собственных идентификаторов вложения будет сложно обновлять, удалять и дедуплицировать.

Индексы payload нужны, чтобы фильтрация по metadata не стала узким местом.

## Варианты MVP По Актуальности

### Вариант 1: Перезапись Последней Версии

Для MVP - мы загружаем последнюю версию страницы - если менялась с последней загрузки - удаляем из Qdrant старую версию, пишем новую.

> **Архитектурное решение бота:** для MVP выбираем перезапись последней версии. Если страница изменилась с последней загрузки, удаляем старые chunks этой страницы из Qdrant и записываем новую версию. В индексе хранится только актуальная версия документа.
> **Ответ на рецензию:** Принято, считаем эту версию как решение для MVP

### Вариант 2: Историчность Через `actual_since` И `actual_to`

ЛИБО

Для MVP - пишем и поддерживаем атрибуты `actual_since` и `actual_to`; соответственно в Qdrant мы только пишем; будет опция найти информацию на конкретную дату.

НО тут проблема, что в старой версии могло не быть инфы, которая есть в текущей, хотя инфа была актуальна и раньше.

> **Рецензия бота:** историчность полезна, но для MVP заметно усложняет модель данных и retrieval-фильтры. Проблема, которую ты отметил, важная: дата появления текста в Confluence не всегда равна дате начала фактической актуальности знания. Поэтому `actual_since` и `actual_to` лучше считать техническими интервалами видимости версии в источнике, а не бизнес-истиной об актуальности факта.

## Рекомендация Для MVP

> **Архитектурное решение бота:** даже при MVP с перезаписью последней версии закладываем metadata для возможной будущей историчности: `source_updated_at`, `ingested_at`, `page_version`, `document_id`, `chunk_id`, `parent_page_id`, `path`. Это сохраняет простой MVP, но не блокирует переход к версии с историческими интервалами.
