# Architecture Draft

Этот документ фиксирует первые наброски архитектуры проекта. Формулировки
предварительные и могут уточняться по мере экспериментов с данными,
retrieval-качеством и агентным workflow.

## RAG Для Данных Confluence

Система взаимодействия с данными, полученными из корпоративного Confluence,
будет основана на RAG. Основная идея: сначала собрать достаточно широкий пул
страниц-кандидатов, затем сузить его более точным ранжированием.

### Первичный Поиск

На первом этапе выполняется гибридный поиск:

- BM25 ищет по лексическому совпадению и хорошо ловит точные термины, названия
  систем, таблиц, топиков и команд.
- Vector search ищет по embeddings и помогает находить семантически близкие
  документы, даже если формулировки запроса отличаются от текста в Confluence.

Ориентир для первичного пула: брать, например, по 50 страниц из BM25 и
vector search. Итого получается до 100 кандидатов до удаления дубликатов.

### Reranking

После первичного поиска применяется cross-encoder reranker. Он прогоняется для
каждой пары «запрос-документ» из списка кандидатов и присваивает каждому
документу новую оценку релевантности. Обычно этот score интерпретируется как
вероятность или степень уверенности, что документ содержит ответ на запрос.

### Финальная Выборка

Страницы-кандидаты сортируются по score reranker-а. В финальный контекст для
генерации ответа попадает top N наиболее релевантных документов.

## HybridRerankerRetriever

Такое решение реализуется через `HybridRerankerRetriever`.

Ретривер должен:

- параллельно запускать BM25 и векторный поиск;
- объединять результаты из обоих retrieval-каналов;
- удалять дубликаты документов;
- применять cross-encoder для финального ранжирования;
- возвращать top-k наиболее релевантных документов.

## Vector Store

В качестве векторной базы используется Qdrant.

Qdrant нужен не только как хранилище embeddings, но и как база для более гибкого
retrieval через Query API, в том числе с одновременной фильтрацией по ключевым
словам, metadata и векторам.

## Agentic RAG Workflow

Проект предполагает agentic RAG с самокорректирующимся циклом.

Базовый workflow:

1. Agent решает, нужен ли retrieval для ответа на вопрос.
2. Если retrieval нужен, агент вызывает tool для получения документов.
3. Retrieve получает документы для запроса и передает их на оценку.
4. Grade Documents оценивает, полезны ли найденные документы, и возвращает
   бинарный сигнал: релевантно или нерелевантно.
5. Generate генерирует ответ, если документы признаны релевантными.
6. Rewrite Query переформулирует вопрос при плохих результатах и возвращает
   управление агенту.

Таким образом, система образует самокорректирующийся цикл:

```text
неудачный поиск -> уточнение запроса -> повторная попытка
```

## Citations And Transparency

Для прозрачности ответа перед пользователем система должна использовать
цитирование источников. Пользователь должен понимать, откуда взята та или иная
информация, особенно если ответ основан на корпоративной документации.

Минимальное требование к ответу: важные утверждения должны быть связаны с
retrieved-контекстом и иметь ссылку на источник.

## Citation Quality Metric

Для оценки качества цитирования используется простая и интерпретируемая метрика:
доля утверждений, имеющих хотя бы один источник.

Метрика напрямую отражает, насколько ответ опирается на retrieved-контекст. При
этом она не оценивает корректность или полноту самих утверждений. Иными словами,
метрика показывает степень опоры ответа на retrieved-контекст, но не гарантирует,
что утверждения верны или полностью покрывают вопрос пользователя.
